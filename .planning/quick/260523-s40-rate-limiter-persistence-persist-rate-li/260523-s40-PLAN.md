<objective>
Replace the in-memory `Map<key, number[]>` rate limiter in
`src/lib/rate-limit.ts` with a SQLite-backed `rate_limit_hits` table so the
5-per-hour cap on `/api/signup` and `/api/manage` survives process restarts
(GitHub Actions deploys ~40s end-to-end currently let an attacker burst
`5 × deploys-per-hour` instead of `5/hour`).

Purpose: Close a real abuse vector. The deploy cadence today means an
attacker who burns a 5/hour budget can wait ~minutes for the next push to
`main` and burst 5 more. The persistent table makes the limit hold across
restarts and across the GitHub Actions deploy cycle.

Output: 3 commit-atomic changes — (1) schema + persistent rate-limit.ts +
hashIp export; (2) call-site wrap with hashIp on signup.ts + manage.ts;
(3) new regression smoke that proves the DB IS the source of truth.

**Locked decisions (per CONTEXT.md — do NOT revisit):**
- **D-01:** Row-per-hit schema `rate_limit_hits(key TEXT, ts INTEGER)` +
  composite index `idx_rlh_key_ts ON (key, ts)`. Counted via
  `SELECT COUNT(*) WHERE key=? AND ts>?`.
- **D-02:** IPs HMAC-hashed with `MAGIC_LINK_SECRET` (sha256, base64url,
  16-char prefix). Emails stay raw.
- **D-03:** Fail-open on DB error + `console.error('[rate-limit] DB error,
  failing open:', err)`.

**Implementation choices documented here (no user re-confirm needed):**
- **MAGIC_LINK_SECRET access pattern: (b) re-read `process.env.MAGIC_LINK_SECRET`
  in rate-limit.ts** with the same prod-only throw guard token.ts:3,16-18 uses.
  Rationale: keeps the two modules independent, avoids a token.ts export
  surface change (no API churn elsewhere), and mirrors the existing idiom
  exactly. The dev fallback string MUST be distinct from token.ts's so an
  accidental cross-purpose use doesn't compare equal under HMAC.
- **Smoke approach: direct-DB pre-seed via better-sqlite3.** The smoke
  computes the same `hashIp` of a synthetic test IP, INSERTs 5 rows into
  `rate_limit_hits` with `ts=now`, then POSTs to `/api/signup` from that IP
  and asserts the response is `303 /?error=rate-limited`. This proves the DB
  is the source of truth without needing to kill+restart the dev server from
  the smoke (which the other smokes don't do — keeps the operator UX
  consistent: "boot dev server, run smoke"). The synthetic IP is RFC 5737
  TEST-NET-1 (192.0.2.x) so it never collides with a real client.
</objective>
