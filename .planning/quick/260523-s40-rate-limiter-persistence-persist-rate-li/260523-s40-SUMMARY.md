# quick-260523-s40: Rate-Limiter Persistence — SUMMARY

**One-liner:** SQLite-backed rate limiter with HMAC-hashed IP keys that survives process restarts and GitHub Actions deploys.

## What this closes

The in-memory `Map<string, number[]>` rate limiter in `src/lib/rate-limit.ts` was reset on every process restart. Since GitHub Actions auto-deploys on push to `main` in ~40s, an attacker who burned the 5/hour budget could wait for the next deploy and burst 5 more — effectively **5 × deploys-per-hour** instead of 5/hour. The persistent `rate_limit_hits` SQLite table makes the limit hold across restarts and across the entire GitHub Actions deploy cycle. The boot-time prune (rows older than 3600s) and opportunistic per-key prune inside `checkRateLimit` keep the table from growing unboundedly.

## Implementation choice records

### (a) MAGIC_LINK_SECRET access pattern

**Choice: re-read `process.env.MAGIC_LINK_SECRET` in rate-limit.ts** with the same prod-only throw guard that `token.ts:3,16-18` uses.

The dev fallback string in `rate-limit.ts` is intentionally distinct from `token.ts`'s `'dev-only-insecure-secret-do-not-use-in-prod'` — `rate-limit.ts` uses `'dev-only-rate-limit-key-do-not-use-in-prod'`. This prevents any accidental cross-purpose HMAC equality. Rationale: keeps the two modules independent, avoids a token.ts export surface change, and mirrors the existing idiom exactly.

### (b) Smoke approach: direct DB pre-seed

**Choice: directly INSERT 5 rows into `rate_limit_hits` via `better-sqlite3`, then POST.**

This proves the DB is the source of truth: the live server (which never saw any HTTP traffic from `192.0.2.99`) immediately rate-limits the 6th POST because `checkRateLimit` reads from the DB, not any in-process state. The approach avoids killing and restarting the dev server from inside the smoke — neither `smoke-signup.mjs` nor `smoke-manage.mjs` does this, maintaining consistent operator UX.

The smoke adds a warmup request to `/api/confirm?token=warmup-ignored` before the pre-flight clean. This is needed because Astro dev mode uses lazy Vite SSR module loading — `db.ts` (and therefore its table migrations) only executes on the first SSR request. The warmup ensures `rate_limit_hits` exists before the smoke tries to write to it.

## Deviations from Plan

None — plan executed exactly as written. The warmup-request addition to the smoke is an implementation detail required by Astro dev server's lazy SSR loading (not architecturally significant; the smoke's core proof is unchanged).
