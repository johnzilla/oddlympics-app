<storage_decision>
<!-- Resolves CONTEXT.md open question #1 — investigated before writing this plan. -->

**Phase 9's "per-purpose token TTL table" is NOT a persisted table.** It's the in-memory `TTL_BY_PURPOSE` constant in `src/lib/token.ts:9-14` plus an idempotent `WHERE unsubscribed_at IS NULL` clause on `markUnsubscribed` (`src/lib/db.ts:185-190`). The "1-year single-use" descriptor in `CLAUDE.md` refers to (a) the 1-year TTL on the unsubscribe token and (b) the fact that a second click of the same unsubscribe link is a no-op because the DB row's `unsubscribed_at` is already set — *the token signature itself is not tracked anywhere*. Manage purpose has no equivalent column to make idempotent.

**Decision: add a new `consumed_tokens` table** per the SQL shape in CONTEXT.md Tactic 3. There is nothing to generalize.

The table is purpose-keyed so future expansion (e.g. consuming `confirm` tokens single-use too) is a parameter change, not a schema change.
</storage_decision>

<implementation_notes>
**Tactic 2 scope (CONTEXT.md open question #3):** `/api/manage` POST returns a 303 redirect with an empty body and no rendered HTML — there is no referer to leak from that response. Set the header on `/manage` (the GET-rendered page) only. One-line confirmation lives in Task 2's verify step.

**Tactic 1 conditional behavior (CONTEXT.md open question #2):** Scrub unconditionally whenever `?token=` is present in the URL — even on the bad-token branch. Honors CONTEXT.md recommendation: an invalid/expired token in history is unwanted noise too, and the conditional logic ("only scrub on success") would require threading the verify result into the inline script.

**Tactic 3 semantic change (acknowledged in CONTEXT.md):** Today a manage link can be re-clicked within 24h. After this change, the second click hits the bad-token branch. The session cookie minted on the first click keeps real users signed in for 30 days — a real user only sees the regression if they (a) used the link, (b) cleared cookies, (c) tried the same link again. Acceptable trade for closing the 24h replay window.

**Tactic 3 TTL janitor:** Prune `consumed_tokens` rows older than 24h opportunistically at module load (alongside the other migration logic in `src/lib/db.ts`). Each row is ~80 bytes; even pessimistic accumulation is well under 8MB, but pruning keeps the table tidy and the index small.
</implementation_notes>

**3a. `src/lib/token.ts` — export a sig extractor.**
Add a named export `extractTokenSig(token: string): string | null` that performs the same string-shape check as `verifyToken` (typeof string, includes '.', non-empty body/sig) and returns the raw sig portion (the part after `.`). This lets `db.ts` key `consumed_tokens.sig` without re-parsing the wire format. Do NOT verify the HMAC — that's `verifyToken`'s job. Returning null means the wire format is malformed; callers should treat that the same as a verify failure.

Signature:
```ts
export function extractTokenSig(token: string): string | null
```

Do NOT change `verifyToken` or `mintToken` signatures — keep the public surface minimal.

**3b. `src/lib/db.ts` — add `consumed_tokens` table, prepared statements, exported wrapper, boot-time prune.**

After the `feature_requests` block (around line 251), add:

```sql
CREATE TABLE IF NOT EXISTS consumed_tokens (
  sig TEXT PRIMARY KEY,
  purpose TEXT NOT NULL,
  consumed_at INTEGER NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_consumed_tokens_consumed_at ON consumed_tokens(consumed_at);
```

The index supports the boot-time prune (range scan by `consumed_at`).

Add prepared statements:
- `const insertConsumedToken = db.prepare<[string, string, number]>('INSERT INTO consumed_tokens (sig, purpose, consumed_at) VALUES (?, ?, ?)');`
- Wrap the verify + consume in a transaction-exported helper:

```ts
// Verify a manage-purpose URL token AND atomically mark it consumed.
// Returns the verified email on success; null if the token is bad,
// expired, wrong purpose, or already consumed.
// Single transaction prevents the TOCTOU race (two concurrent clicks
// both passing verify before either inserts).
export function consumeManageToken(token: string): { email: string } | null {
  const sig = extractTokenSig(token);
  if (!sig) return null;
  const verified = verifyToken(token, 'manage');
  if (!verified) return null;
  const now = Math.floor(Date.now() / 1000);
  try {
    insertConsumedToken.run(sig, 'manage', now);
    return verified;
  } catch (err: unknown) {
    // SQLITE_CONSTRAINT_PRIMARYKEY → already consumed.
    if (err instanceof Error && 'code' in err &&
        (err as { code: string }).code === 'SQLITE_CONSTRAINT_PRIMARYKEY') {
      return null;
    }
    throw err;
  }
}
```

Note: `db.transaction()` is not strictly required — the single INSERT on a PRIMARY KEY constraint is itself atomic, and the verify call is read-only against the DB (it only does HMAC + JSON). Documenting "single transaction" in the comment is fine because the INSERT IS the atomic act; a concurrent click reaches the same INSERT and one of them fails the PK constraint. (If review prefers an explicit `db.transaction(() => {...})` wrapper for documentation clarity, that's also acceptable — but it adds no semantic.)

Imports: `import { verifyToken, extractTokenSig } from './token';` at the top of `db.ts` (adjacent to the existing imports). This creates a one-way dependency `db.ts` → `token.ts`, which already exists implicitly via `manage.astro` using both — no cycle risk.
