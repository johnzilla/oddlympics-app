<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| /api/signup → sendMagicLink | The `referralCode` argument arrives from `row.referral_code` (DB-issued 8-char [a-z0-9] code from `generateReferralCode`, written under a UNIQUE index). Not user-controlled. |
| sendMagicLink → outbound email body | The composed HTML and plaintext bodies are sent via Resend to the user's confirmed email address. No third-party content interpolation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-07 | Tampering | HTML injection via teamHuman interpolation in the new share `<p>` | accept | `teamHuman` is `teamLabel(team)` (`teams.ts:16-18`) returning a value from the checked-in `references/teams.json` allow-list (48 labels, no special chars). The existing `:51` line already interpolates `${teamHuman}` inside `<strong>` without escaping — this plan follows the existing convention. No user input reaches the email body. |
| T-14-08 | Tampering | URL injection via referralCode interpolation in the share `<a href>` | mitigate | `referralCode` is constrained by `generateReferralCode` to `[a-z0-9]{8}` and by the DB UNIQUE index. No `"`, `>`, `<`, `&`, or space can reach the href attribute. Defensive: the code is also interpolated unescaped in `:24` (the confirm-link URL) using the same trust assumption. If an attacker somehow controlled `referralCode` (they do not — it is DB-issued), the worst case is a malformed URL, not a script tag. |
| T-14-09 | Information Disclosure | share line leaking PII in email body | accept | The share URL contains only the user's own public referral code — a value designed to be shared. No email, no name, no IP, no token. The Web Share API payload and the email body carry the same public string. |
| T-14-10 | Repudiation | dev fallback logs share URL to stdout / journald | accept | Dev fallback fires only when `RESEND_API_KEY` is unset, which never happens in production (`email.ts:12-14` throws on prod boot if missing). Logging the share URL alongside the already-logged magic-link URL adds no incremental disclosure risk — the magic-link URL is far more sensitive (it grants confirmation) and is already logged in dev. |
| T-14-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced this plan — zero install surface. |
</threat_model>
