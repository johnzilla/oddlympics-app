<objective>
Add the share UI — a visible readonly URL input + a Share/Copy button with Web Share API feature-detection + clipboard fallback — to `/pending`, `/confirmed`, and the signed-in branch of `/manage`. The three pages share the same UX (heading + URL + button + "Copied!" flash) but differ in how they obtain the referral code: the two prerendered pages read `?rc=` from the URL (placed there by Plan 14-02); `/manage` reads `user.referral_code` directly from its server frontmatter (no URL-param dance).

Purpose: SHARE-01 — share prompt on `/pending`, `/confirmed`, `/manage`. SHARE-03 — native share sheet with copy-link fallback. SHARE-04 — message names the user's team. D-05 / D-06 / D-15 / D-16 / D-17 lock the per-page rules and the inline-component decision.

Implementation note: per CONTEXT.md Discretion, the share UI is inline-duplicated markup (not a shared `ShareCard.astro` component). Rationale: the per-page CSS is shared via Layout.astro CSS variables (--accent, --accent-soft, --surface, --line, --mono, --pad), and each page has at most ~15 lines of share-card markup + a tightly scoped inline script — under the threshold that justified the Layout.astro extraction in v2.0 Phase 11. If any of the three pages diverges visually later, a `ShareCard.astro` extraction can follow (same pattern that pulled Layout.astro forward).

Output: updated `src/pages/pending.astro`, `src/pages/confirmed.astro`, and `src/pages/manage.astro` — each with share-card markup + share-button logic, plus per-page `<style>` additions reusing Layout.astro tokens.
</objective>

The Web Share API feature-detection + clipboard fallback pattern (locked by D-06):
1. On button click, `if (typeof navigator !== 'undefined' && typeof navigator.share === 'function') { navigator.share({ text: shareText, url: shareUrl }).catch(function(e) { if (e && e.name !== 'AbortError') { /* fall through to clipboard */ } }); }`.
2. Else (or on non-AbortError share failure): `navigator.clipboard.writeText(shareText + '\n' + shareUrl).then(function() { button.textContent = 'Copied!'; setTimeout(function() { button.textContent = 'Share'; }, 1500); }).catch(function() { /* readonly input is still selectable by hand */ });`.
3. All wrapped in `try { ... } catch {}` per CONVENTIONS.md.

Share-text composition in the three pages (mirroring shareText's contract but in inline JS, since shareText is a server-side TS function and prerendered pages cannot import server modules at runtime):
- The prerendered pages BUILD the share text inline in JS: `const teamLabelText = teamFromUrl || ''; const shareTextStr = "I'm following " + (teamLabelText ? humanLabelForSlug(teamLabelText) : 'my team') + " — get your team's World Cup kickoff alerts.\\n" + shareUrl;`. The team-to-label resolution in the browser uses a small inline lookup table (the 48 slug→label pairs from references/teams.json) embedded into the page at BUILD time via a small Astro-frontmatter `JSON.stringify(...)` block — see action step (b1).
- `/manage` (server-rendered) can call `shareText(teamLabel(user.team), shareUrl)` directly in the frontmatter and pass the result to the inline script via a `data-share-text` attribute or a `JSON.stringify` injection (see action step (d2)).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL `?rc=` and `?team=` params on /pending and /confirmed | Both query values are fully attacker-controllable (anyone can craft `/pending?email=victim@example.com&rc=<script>alert(1)</script>&team=<svg>`); the inline script writes them into DOM nodes |
| /manage `user.referral_code` and `user.team` | Server-rendered from the verified session; the row was created by the verified user's own signup → `referral_code` is DB-issued, `team` is from `VALID_TEAMS` allow-list. Not attacker-controlled at this trust boundary. |
| clipboard write | `navigator.clipboard.writeText` writes a server-or-script-composed string; never reads from clipboard. No clipboard hijack vector. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-11 | Tampering | `?rc=` URL param XSS via attribute injection on /pending and /confirmed | mitigate | The rc value is gated by `/^[a-z0-9]{8}$/.test(rc)` BEFORE any DOM write — a non-matching value causes the script to return early and leave the share card hidden. The `share-url` input value is assigned via DOM API (`element.value = ...`), never via `setAttribute` or template-string concatenation into HTML. The gate is the security boundary; the DOM API assignment is defense in depth. |
| T-14-12 | Tampering | `?team=` URL param XSS via textContent | mitigate | The team value is looked up in the `TEAM_LABEL_JSON` map (the 48-entry allow-list from references/teams.json embedded at build time); unknown values resolve to the empty string and fall back to the generic blurb. | T-14-13 | Tampering | share-url injection into `<input value=>` via setAttribute path | mitigate | Code uses `inputEl.value = shareUrl` (DOM property assignment), not `inputEl.setAttribute('value', shareUrl)` or template-literal HTML concatenation. The rc has already passed the `[a-z0-9]{8}` gate. Both layers are belt-and-braces. |
| T-14-14 | Information Disclosure | share URL exposed in history / address bar | accept | Same disposition as T-14-03 in Plan 14-02 — the referral code is a public identifier by design. |
| T-14-15 | Tampering | clipboard hijack via clipboard READ | accept | Code calls only `navigator.clipboard.writeText` (write-only). No `readText` anywhere. The Permissions Policy in the browser also gates clipboard read separately from write. |
| T-14-16 | Spoofing | self-XSS via teamLabel containing `<script>` from a poisoned references/teams.json | accept | `references/teams.json` is checked into the repo and code-reviewed.
