## Phase Goal

**As a** subscriber receiving a (future) notification email from oddlympics, **I want to** click a one-click unsubscribe link and have it remove me from the list, **so that** I have a working, RFC 8058-compliant unsubscribe path before any production email campaign runs.

<objective>
Build the complete unsubscribe vertical slice: schema column, token purpose claim, GET endpoint, status page, and an RFC 8058 header helper for outbound non-confirmation email. Per CONTEXT.md `<resolve_open_questions>`, scope is **option (a) — full scaffolding**: every piece end-to-end so the success criterion ("a user who clicks an unsubscribe link is removed from future mailings") can be tested via curl in Phase 1, and so Phase 3's `sendNotification` can wire up trivially by calling `buildUnsubscribeHeaders(email)` at send-time.

Per D-07 (CONTEXT.md): the existing `sendMagicLink` (the double-opt-in confirmation) is EXEMPT from the unsubscribe requirement under CAN-SPAM and is NOT modified to add List-Unsubscribe headers. The helper is ready and unit-testable; Phase 3 calls it from notification senders.

Per D-05 (CONTEXT.md): soft delete via `unsubscribed_at` column; future notification queries filter `WHERE unsubscribed_at IS NULL`. Hard delete deferred.

Per D-08 (CONTEXT.md): RFC 8058 means `List-Unsubscribe: <https://oddlympics.app/api/unsubscribe?token=...>` AND `List-Unsubscribe-Post: List-Unsubscribe=One-Click`. Both headers are required for Gmail's 2024 bulk-sender requirements.

**Open question resolutions documented inline:**

1. **`unsubscribed_at` column type:** Override D-05's "TEXT" → use **`INTEGER`** to match the existing `confirmed_at` and `created_at` storage convention (db.ts:20-21, populated via `strftime('%s','now')` returning unix epoch seconds). PATTERNS.md flagged this; consistency with the in-tree convention is more valuable than literal D-05 wording. The CONTEXT.md user decision still gates the WHAT (a soft-delete timestamp column); type choice is Claude's Discretion within the schema-design domain.

2. **`mintToken` signature:** Use an **options object**: `mintToken(email: string, opts?: { purpose?: 'confirm' | 'unsubscribe' })`. Backward-compatible with the single existing call site (`signup.ts:85` — `mintToken(rawEmail)` — no change required). Slightly more flexible than a positional arg if a future plan wants to override TTL per purpose.

3. **HARDEN-02 scope:** Full scaffolding — DB column, /api/unsubscribe endpoint, /unsubscribed page, signed-token purpose claim, RFC 8058 helper. The helper is exported and ready; Phase 3 wires it.

Output: 5 files modified or created. Phase 1 ends with a working unsubscribe path testable end-to-end via curl: mint a token with purpose='unsubscribe' (from a node REPL or a transient test fixture), hit `/api/unsubscribe?token=...`, observe 303 to `/unsubscribed?status=ok`, observe `unsubscribed_at` populated in the DB, click again, observe 303 to `/unsubscribed?status=already`.
</objective>

export function verifyToken(token: string): { email: string } | null {
  if (typeof token !== 'string' || !token.includes('.')) return null;
  const [body, sig] = token.split('.');
  if (!body || !sig) return null;

  const expected = sign(body);
  const a = Buffer.from(sig);
  const b = Buffer.from(expected);
  if (a.length !== b.length || !timingSafeEqual(a, b)) return null;

NEW:
```
export function verifyToken(
  token: string,
  expectedPurpose?: 'confirm' | 'unsubscribe',
): { email: string } | null {
  if (typeof token !== 'string' || !token.includes('.')) return null;
  const [body, sig] = token.split('.');
  if (!body || !sig) return null;

  const expected = sign(body);
  const a = Buffer.from(sig);
  const b = Buffer.from(expected);
  if (a.length !== b.length || !timingSafeEqual(a, b)) return null;

  let payload: Payload;
  try {
    payload = JSON.parse(fromB64url(body).toString('utf8'));
  } catch {
    return null;
  }
  if (typeof payload.email !== 'string' || typeof payload.exp !== 'number') return null;
  if (Math.floor(Date.now() / 1000) > payload.exp) return null;

  if (expectedPurpose) {
    // Legacy tokens (minted before D-06) have no purpose field; treat them as 'confirm'
    // for backward compatibility per D-04 (no proactive invalidation of in-flight tokens).
    const tokenPurpose = payload.purpose ?? 'confirm';
    if (tokenPurpose !== expectedPurpose) return null;
  }

  return { email: payload.email };
}

export function buildUnsubscribeHeaders(email: string): {
  'List-Unsubscribe': string;
  'List-Unsubscribe-Post': string;
} {
  const token = mintToken(email, { purpose: 'unsubscribe' });
  const url = `${SITE_URL}/api/unsubscribe?token=${encodeURIComponent(token)}`;

}

import type { APIRoute } from 'astro';
import { verifyToken } from '../../lib/token';
import { markUnsubscribed, getByEmail } from '../../lib/db';

export const prerender = false;

export const GET: APIRoute = async ({ url }) => {
  const token = url.searchParams.get('token');
  if (!token) return redirect('/unsubscribed?status=bad-token');

  const result = verifyToken(token, 'unsubscribe');
  if (!result) return redirect('/unsubscribed?status=bad-token');

  const updated = markUnsubscribed.get(result.email);
  if (updated) return redirect('/unsubscribed?status=ok');

  // Either already unsubscribed or no such row. Idempotent: still send to /unsubscribed.
  const existing = getByEmail.get(result.email);
  if (existing) return redirect('/unsubscribed?status=already');
  return redirect('/unsubscribed?status=unknown');
};

function redirect(to: string): Response {
  return new Response(null, { status: 303, headers: { Location: to } });
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| email inbox → click → /api/unsubscribe | Unsubscribe token is a bearer credential. Anyone in possession of the URL can unsubscribe the embedded email. |
| internet → GET /api/unsubscribe | Untrusted query param `?token=` crosses here. |
| outbound email → recipient mail client | RFC 8058 headers must be exactly correct or Gmail will not surface the native unsubscribe button. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | unsubscribe token forgery | mitigate | HMAC-SHA256 signed via `MAGIC_LINK_SECRET`; `timingSafeEqual` after length check (token.ts:42-44, preserved). Forging requires the secret. |
| T-04-02 | Tampering | token modification (purpose claim swap) | mitigate | Purpose claim is inside the signed payload body; modifying it invalidates the HMAC. The `expectedPurpose` check in verifyToken (this plan) ALSO prevents a valid confirm-purpose token from being replayed against /api/unsubscribe. |
| T-04-03 | Information Disclosure | enumeration via token guessing | mitigate | b64url payload + 256-bit HMAC over a 32-byte secret. Brute-force search space is ~2^256. Practically impossible. |
| T-04-04 | Repudiation / Replay | reuse of unsubscribe token | mitigate | Row-level idempotency: `markUnsubscribed WHERE unsubscribed_at IS NULL` makes the second click a no-op. Plus 24h TTL bounds the window. |
| T-04-05 | Cross-Site Request Forgery (T) | GET-based unsubscribe vs. RFC 8058 POST | accept | RFC 8058 prefers `List-Unsubscribe-Post: List-Unsubscribe=One-Click` (POST), which Gmail uses internally. The visible link in the email is GET; this is conventional and matches all major sender implementations. The unsubscribe action is benign-by-design (sets a flag) so a CSRF attempt at worst unsubscribes the holder of the token, who can re-subscribe. Risk accepted. |
| T-04-06 | Information Disclosure | rendered status page revealing email existence | mitigate | `/unsubscribed?status=unknown` and `?status=already` render distinct copy; an attacker enumerating /unsubscribed?status=... cannot infer membership without a valid signed token, because the route handler only branches on the result of token verification — direct visits to /unsubscribed render the COPY map at random regardless of DB state. `<meta name="robots" content="noindex">` keeps status pages out of search indexes. |
| T-04-07 | Denial of Service | flood of /api/unsubscribe with invalid tokens | accept | Each request is O(1) (HMAC verify + early-return). Caddy edge can be rate-limited if it becomes a problem; v1 scale does not warrant pre-emptive measures. |
| T-04-08 | Elevation | header injection via email parameter | mitigate | `buildUnsubscribeHeaders` only reads the email into the token payload, then encodes the resulting token via `encodeURIComponent`. The email is never interpolated directly into a header value. |
</threat_model>
