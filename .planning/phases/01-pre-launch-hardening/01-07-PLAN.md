**As a** site operator one droplet failure away from losing the email list, **I want to** have an automated daily backup of /var/lib/oddlympics/oddlympics.db pushed to a separate cloud (Backblaze B2), with a documented and tested restore procedure, **so that** a droplet loss does not lose the captured emails and a corruption incident has a known-good recovery path.

- Scheduler: **systemd timer** (oddlympics-backup.service + .timer) running as `oddlympics` user, journal logs.
- Tooling: **rclone** with B2 backend. Application key scoped to a single bucket (least privilege).
- Retention: 30 days daily + 12 weeks weekly via B2 lifecycle rules (preferred — set up in the B2 console; the script just keeps writing daily snapshots).
- Credentials: `/etc/oddlympics-backup.env` (root-owned, group-readable by oddlympics), separate from `/etc/oddlympics.env` so the app cannot read the backup key.

**Step C: Create `deploy/oddlympics-backup.sh`** with these exact contents:

```
#!/usr/bin/env bash
# oddlympics SQLite backup -> Backblaze B2.
# Run by oddlympics-backup.service; do not invoke directly without the
# matching environment file (/etc/oddlympics-backup.env).
set -euo pipefail

DB_PATH="${DATABASE_PATH:-/var/lib/oddlympics/oddlympics.db}"
CACHE_DIR=/var/cache/oddlympics-backup
TS=$(date -u +%Y-%m-%dT%H-%M-%SZ)
SNAPSHOT="${CACHE_DIR}/oddlympics-${TS}.db"
REMOTE="${RCLONE_REMOTE:-b2}:${B2_BUCKET}/daily/oddlympics-${TS}.db"

: "${B2_ACCOUNT_ID:?B2_ACCOUNT_ID required (see /etc/oddlympics-backup.env)}"
: "${B2_APPLICATION_KEY:?B2_APPLICATION_KEY required}"
: "${B2_BUCKET:?B2_BUCKET required}"

mkdir -p "$CACHE_DIR"

# Use SQLite's online backup API (handles WAL, atomic, no lock contention).
sqlite3 "$DB_PATH" ".backup '${SNAPSHOT}'"

# Quick integrity check before uploading
if ! sqlite3 "$SNAPSHOT" "PRAGMA integrity_check" | grep -qx "ok"; then
  echo "[oddlympics-backup] integrity check FAILED for $SNAPSHOT" >&2
  rm -f "$SNAPSHOT"
  exit 1
fi

# rclone reads B2 credentials from RCLONE_CONFIG_<remote>_<key> env vars.
# We set them via the EnvironmentFile so no rclone.conf on disk is required.
export RCLONE_CONFIG_B2_TYPE=b2
export RCLONE_CONFIG_B2_ACCOUNT="$B2_ACCOUNT_ID"
export RCLONE_CONFIG_B2_KEY="$B2_APPLICATION_KEY"

rclone copyto "$SNAPSHOT" "$REMOTE" --quiet

# Local cache cleanup: keep the last 7 local snapshots so the operator can
# restore quickly without round-tripping to B2.
ls -1t "$CACHE_DIR"/oddlympics-*.db 2>/dev/null | tail -n +8 | xargs -r rm -f

echo "[oddlympics-backup] uploaded $SNAPSHOT -> $REMOTE"
```

This script: uses the SQLite online backup API (safe against a running app process); runs an integrity check before uploading (aborts on failure); uses rclone env-config (no rclone.conf on disk); uploads to `daily/oddlympics-<timestamp>.db` in the bucket; D-13 retention is enforced via B2 lifecycle rules in the console (per Task 4 user_setup), not in the script; keeps a small local cache (last 7 snapshots) for fast emergency restore.

## Backups and restore

The droplet runs `oddlympics-backup.timer` daily at **03:00 UTC**, firing
`oddlympics-backup.service` (a systemd oneshot) which:

1. Takes a SQLite snapshot via `sqlite3 .backup` (atomic, WAL-safe).

3. Uploads the snapshot to `b2:$B2_BUCKET/daily/oddlympics-<timestamp>.db`.
4. Keeps the last 7 snapshots locally in `/var/cache/oddlympics-backup` for fast emergency restore.

**Retention policy** (configured as B2 lifecycle rules in the bucket settings):

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| droplet -> B2 | rclone uploads cross the public internet. Backups contain the captured email list (PII per GDPR). |
| /etc/oddlympics-backup.env -> backup process | Credentials must be readable by the oddlympics user but not the app process (D-15 separation). |
| B2 bucket -> any party with the application key | Bucket contents are an attractive target (real email list). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | credential leak via /etc/oddlympics-backup.env readability | mitigate | Mode 640, owner root, group oddlympics. App process runs as oddlympics but does NOT load this env file (the systemd unit oddlympics.service reads only /etc/oddlympics.env). Per D-15, the separation is the mitigation. |
| T-07-02 | Information Disclosure | leaked B2 application key reveals all backups | mitigate | Key is scoped to a single bucket via the B2 console (per D-12 user_setup task). Compromise lets the attacker read/write THAT bucket only — not all buckets, not the account. Rotation procedure: generate new key in console, update /etc/oddlympics-backup.env, revoke old key. |
| T-07-03 | Tampering | attacker overwrites snapshots in the bucket | mitigate (partial) | The backup user can write to the bucket, so a compromised droplet COULD upload corrupted snapshots. Mitigation: B2 lifecycle rules retain the last 30 daily snapshots (D-13), so an attacker would need to corrupt every one for 30 days running before all good copies are gone. Future improvement (deferred to v1.x): use B2 Object Lock for write-once retention; not in Phase 1 scope. |
| T-07-04 | Denial of Service | rclone hangs / B2 outage breaks the timer | accept | The script has no built-in timeout; on B2 outage, the systemd unit stays in `activating` state until the next firing, which simply runs on top. Failed runs are logged to journald and surface via `systemctl status`. Operator-monitored. For v1 scale this is acceptable. |
| T-07-05 | Tampering | snapshot integrity in transit | accept (TLS) | rclone uses HTTPS to api.backblazeb2.com; TLS handles transit integrity. B2 verifies SHA1 server-side. |
| T-07-06 | Information Disclosure | snapshot leaks email PII via cache dir readability | mitigate | /var/cache/oddlympics-backup is mode 750, owner oddlympics. Other users on the droplet (only root and deploy exist) cannot read. ProtectSystem=strict in the service unit prevents writes outside the explicit ReadWritePaths. |
| T-07-07 | Repudiation | which key uploaded which snapshot | accept | B2 application keys carry an audit trail in the B2 console. For v1 with one droplet and one key, this is sufficient. |
| T-07-08 | Elevation | backup user escalates via the new service unit | mitigate | The service unit clones the hardening flags from oddlympics.service: NoNewPrivileges=true, ProtectSystem=strict, ProtectHome=true, ProtectKernel*=true, RestrictAddressFamilies, etc. The backup user has no sudo rights. The systemd timer fires the unit directly — no sudo path involved. |
</threat_model>
