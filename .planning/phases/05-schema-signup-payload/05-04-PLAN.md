<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → POST /api/signup | Untrusted form data. The pre-flight chain (Origin → honeypot → email regex → rate limit → team/tz allow-list) is the trust gate. |
| browser tz string → server log | Untrusted raw input echoed in a log line; CRLF injection could split log entries. |
| Set-based allow-list lookup | No injection vector — `Set.has()` is a value comparison; the value never reaches SQL except through prepared-statement parameters. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | T (Tampering) | Bad `timezone` input embeds CRLF or control chars; the tz-fallback log line could be split into a forged second log entry | mitigate | The log line uses `JSON.stringify(rawTz)` which escapes control characters (CRLF becomes `\r\n` in the printed string). Same treatment applied to the bad-team log line for `rawTeam`. |
| T-05-10 | T (Tampering) | SQL injection via `team` slug | mitigate | Two layers: (1) `VALID_TEAMS.has(rawTeam)` Set lookup rejects anything not in the 48-element allow-list before the value reaches SQL; (2) the prepared statement passes `team` as a bound parameter, not interpolated string. Even if (1) is bypassed by code error, (2) makes injection impossible. |
| T-05-11 | T (Tampering) | SQL injection via `timezone` (no allow-list match → falls back to a constant) | mitigate | The fallback path replaces `rawTz` with `FALLBACK_TZ` (a literal constant); the prepared statement only ever sees either an IANA zone from the Set or the hardcoded fallback. Bound parameter is the second layer. |
| T-05-12 | S (Spoofing) | Bot bypasses honeypot but hits team allow-list rejection | mitigate (defense in depth) | The team allow-list is the SECOND filter after honeypot. A bot that fills the honeypot still gets silent fake-success per existing teaser behavior (lines 55-59 preserved). A bot that skips the honeypot but submits `team=fake_team` is rejected at the allow-list — no row written, no email sent. Bots requiring a real-team-name dictionary attack scale linearly with their slug-guessing rate; rate limit already bounds them to one POST per minute per IP per email. |
| T-05-13 | I (Info disclosure) | Log line includes raw user email | accept | The existing handler already logs the email in `[signup] db error` and `[signup] email error` lines (CONVENTIONS.md §Logging). Operator-only access via `journalctl -u oddlympics` per CLAUDE.md "Production" hardening. No change to disclosure surface. |
