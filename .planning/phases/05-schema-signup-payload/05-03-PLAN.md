<objective>
Land the `vip_signups` schema migration that retires v1's `selected_teams TEXT` (JSON array of football-data team IDs) and replaces it with `team TEXT` (single snake_case slug from `references/teams.json`). Per CONTEXT.md D-01 this is the first non-additive change in the project — every prior migration only added columns. It is safe because there is exactly one row in the live DB (the operator's), and that row will be re-set manually post-deploy via Plan 05's updated `/api/save-selection` or via a one-off SQL set.

Mechanics:
1. Pre-deploy backup: `scripts/backup-pre-05.mjs` copies `data/oddlympics.db` → `data/oddlympics.db.pre-05.bak` (operator runs on droplet before pushing the deploy with the migration).
2. Idempotent migration block in `src/lib/db.ts`: add `team` column (existing `pragma_table_info` probe pattern), then drop `selected_teams` (same probe pattern — SQLite has no `DROP COLUMN IF EXISTS`).
3. Update `VipSignup` type to remove `selected_teams` and add `team`.
4. Rewrite the `setSelection` prepared statement to write `team` instead of `selected_teams`. Param tuple stays `<[string, string, string]>` (slug, tz, email).

SQLite version requirement: ≥ 3.35 for `ALTER TABLE … DROP COLUMN`. Local dev SQLite is 3.51 (confirmed at plan time). Droplet Ubuntu shipped SQLite is well above the floor. The migration block aborts cleanly with a clear error if runtime ever falls below 3.35.

This plan does NOT update the downstream consumers (`send-kickoff-notifications.mjs`, `schedule.astro`, `save-selection.ts`) — that's Plan 05 (same wave, no file overlap). Both plans must merge to main before deploy so the live droplet doesn't boot into a half-migrated state.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| module load → SQLite DDL | The db.ts import runs schema-changing DDL at process start. A bad migration aborts boot — the systemd unit will restart on failure (CLAUDE.md mentions ~2s `RestartSec`), creating a brief outage if the migration is broken. |
| operator → backup script | Operator runs the backup before deploy; forgetting it is a procedural risk, not a code one. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | D (Denial of service) | `DROP COLUMN` fails on a SQLite < 3.35 runtime; module-load throw → systemd restart loop | mitigate | Explicit version assert at the top of the probe block aborts with a clear error before the DDL fires. Operator gets a `journalctl -u oddlympics` line naming the SQLite version, can roll back the deploy or upgrade SQLite. |
| T-05-06 | T (Tampering) | Lost operator row data (the single live row) when `selected_teams` is dropped | mitigate | (a) Pre-deploy backup script (Task 1) snapshots the DB file; (b) DigitalOcean Backups (v1 HARDEN-05 replacement, enabled 2026-05-10) is the secondary recovery floor; (c) `team` column is added before `selected_teams` is dropped, so the next deploy that ships Plan 05's `/api/save-selection` rewrite gives the operator a UI to set their `team` slug. |
| T-05-07 | I (Info disclosure) | The backup file (`.pre-05.bak`) sitting alongside the live DB after deploy — slightly higher attack-surface footprint on the droplet | accept | The droplet is single-tenant operator-owned, `data/` is restricted to the `oddlympics` user via systemd's `ProtectSystem=strict` + group permissions (per CLAUDE.md "hardening flags"). A second SQLite file with the same permissions adds no meaningful disclosure risk. The script's header comment instructs the operator to remove the backup once Phase 5 is verified stable. |
| T-05-08 | E (Elevation of privilege) | Migration block runs at every module load; a malicious actor who can edit `src/lib/db.ts` already has full app control | accept | This is the existing trust model — `src/` is in the deploy artifact, no privilege escalation here. |
</threat_model>
