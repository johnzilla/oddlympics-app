<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → POST /api/save-selection | Authenticated (token or session cookie) request. Trust gate is the auth check, not the input. |
| `team_ids` form input → teams.id → teams.slug lookup | Untrusted integer input; resolved to slug via DB lookup (no injection vector — bound param). |
| Plain SQLite read in schedule.astro frontmatter | Astro frontmatter is server-side; user.team resolves via prepared statement bound param. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-15 | T (Tampering) | A user submits `team_ids=999999` (non-existent) — falls through the regex (`/^\d{1,8}$/`), the slug lookup returns no row, the loop continues with `teamSlug` still null, eventually hits the "no valid team" branch and redirects with `too-many` status | mitigate | The slug-lookup-OR-skip loop pattern means a bad ID never reaches the prepared statement with a fabricated value; only resolved slugs land in the DB. The redirect-on-empty branch surfaces the bad input to the user. |
| T-05-16 | T (Tampering) | A user submits 1000 `team_ids` entries hoping to DoS the slug lookup | mitigate | The for-loop breaks on FIRST valid resolution (`break;` after setting `teamSlug`). Worst case is 1000 iterations of regex + integer parse + a single SQLite point-query — total ~5 ms. Rate limit on the route (none currently exists but the same droplet-level Caddy rate limit applies) bounds aggregate cost. Existing TEAM_ID_RE (`/^\d{1,8}$/`) caps each ID at 8 digits. |
| T-05-17 | D (DoS) | Slug-lookup query has no index on `teams.slug` | accept | The `teams` table has 48 rows total. Even without an index, a SELECT WHERE slug = ? is a 48-row scan — sub-millisecond. If/when the table grows (other tournaments), Phase 9 or a future plan adds an index. Not Phase 5's concern. |
| T-05-18 | I (Info disclosure) | Frontmatter slug lookup in `schedule.astro` exposes a slug → id mapping via response timing? | accept | The mapping is public (slugs come from a committed `references/teams.json`, ids come from football-data.org public API). No secret. |
| T-05-19 | T (Tampering) | A returning user submits a SQL-injection-flavored slug via custom client | mitigate | All three consumers use prepared statements with bound parameters. No string interpolation. (CONVENTIONS.md §"Security posture": "DB writes parameterized via prepared statements — no string interpolation.") |
