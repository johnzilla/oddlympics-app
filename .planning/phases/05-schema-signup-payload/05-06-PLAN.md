<objective>
Ship the Phase 5 end-to-end verification: a single `scripts/smoke-signup.mjs` that boots against a running dev server, drives `POST /api/signup` through the seven cases mandated by CONTEXT.md D-07 + MILESTONE-consumer-landing.md Phase 1, and exits 0 only when all seven pass. This is the AC9 + AC12 + AC2 (backend portion) evidence for Phase 5.

No new test framework. Per D-07 the codebase has no vitest/jest and Phase 5 is not the place to introduce one — `scripts/smoke-*.mjs` is the established pattern (mirrors `scripts/launch-blast.mjs`, `scripts/send-kickoff-notifications.mjs`, and the Plan 02/03 backfill+backup scripts).

Cases (verbatim from MILESTONE-consumer-landing.md Phase 1):
1. Valid submission (team + email + tz all good) → 303 to `/pending?email=...`; row persists with the right fields.
2. Missing team → 303 to `/?error=bad-form`; no row.
3. Invalid team (`team=fake_team`) → 303 to `/?error=bad-form`; no row. **AC9.**
4. Missing tz (empty string) → 303 to `/pending`; row persists with `tz='America/New_York'`; log line `[signup] tz-fallback ...` visible in stdout (out-of-band assertion).
5. Invalid tz (`timezone=Foo/Bar`) → same as #4 (fallback path).
6. Honeypot triggered (`website=spam`) → 303 to `/pending` (silent fake success); no row. **AC12.**
7. Rate limit triggered (>N requests from same IP in window) → 303 to `/?error=rate-limited`.

Plus a static assertion of the FORM-02 explicit slug list against `references/teams.json` (AC2 backend portion).

The script seeds a fresh email per case (e.g., `smoke-<case>-${Date.now()}@example.com`) so cases are idempotent and don't collide with prior runs.
</objective>

<interfaces>
The smoke script consumes the post-Plan-04 contract of `POST /api/signup`:

```
form fields:
  email (required, string, ≤254 chars, regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/)
  team (required, string, must be in references/teams.json slugs)
  timezone (optional, string, must be in Intl.supportedValuesOf('timeZone'); else falls back to 'America/New_York')
  requested_sport (optional, default 'world_cup')
  website (HONEYPOT; if non-empty: 303 to /pending, no row)

response codes:
  303 Location: /pending?email=... — success path
  303 Location: /?error=bad-form — invalid team, missing team, malformed formData
  303 Location: /?error=bad-email — bad email regex or >254
  303 Location: /?error=bad-origin — Origin check failed
  303 Location: /?error=rate-limited — rate limit triggered
  303 Location: /?error=email — sendMagicLink threw
  303 Location: /?error=server — DB write threw
  303 Location: /pending (no email param) — honeypot triggered

env that affects the test:
  PUBLIC_SITE_URL — must match Origin header to pass originOk; default http://localhost:4321
  Rate limit constants — see src/lib/rate-limit.ts (read at task time)
```

Test environment: the script assumes a dev server running at `process.env.SMOKE_BASE_URL ?? 'http://localhost:4321'`. The operator boots it via `npm run dev` (or `npm run build && npm run serve`) in a separate terminal before running `node scripts/smoke-signup.mjs`. The script does NOT fork the server — it just fetches against an already-running one.

    - `import { readFileSync } from 'node:fs';`
    - `import { resolve } from 'node:path';`

    **Helper: `post(form, extraHeaders = {})`:**
    Build a `URLSearchParams` from the form object, POST to `${BASE}/api/signup` with `Content-Type: application/x-www-form-urlencoded` and `Origin: ${BASE}` (so the originOk check passes).     **Helper: `runCase(name, fn)`:** wraps a single case in try/catch (try/catch is permitted in scripts — it's not the rule against try/catch around things that can't reasonably throw; here, network errors and DB errors are real). On exception logs `[smoke] FAIL ${name} (exception) ${err.message}`. Returns boolean. Accumulates pass/fail counts in module-level locals.

    Read `references/teams.json` via `readFileSync` + `JSON.parse`. Assert:
    - Length is 48.
    - Every slug in the FORM-02 explicit set is present: `['united_states','south_korea','ivory_coast','dr_congo','cape_verde','bosnia','czech_republic','new_zealand','saudi_arabia','south_africa']`.
    Log `[smoke] PASS AC2-teams-json (48 entries, FORM-02 slugs verified)` or fail with the missing slug.

    Email = `smoke-bad-team-${Date.now()}@example.com`. POST with `team=fake_team`, `timezone=Europe/London`. Assert: status=303, location is `/?error=bad-form`. No row written.

    **Case 4 — missing tz (falls back):**
    Email = `smoke-miss-tz-${Date.now()}@example.com`. POST with `team=brazil`, `timezone=''`. Assert: status=303, location starts with `/pending?email=`. After: row has `timezone='America/New_York'` (the fallback), `team='brazil'`.

    **Case 5 — invalid tz (falls back):**
    Email = `smoke-bad-tz-${Date.now()}@example.com`. POST with `team=france`, `timezone=Foo/Bar`. Assert: same as Case 4 — 303 to /pending, row has `timezone='America/New_York'`, `team='france'`.

    Email = `smoke-honeypot-${Date.now()}@example.com`. POST with `team=germany`, `timezone=Europe/Berlin`, `website=evil-bot`.     **Case 7 — rate limit:**
    Read `src/lib/rate-limit.ts` at task execution time to learn the burst limit and window. The script needs to: (a) pick a single email + IP (set `X-Forwarded-For: 192.0.2.42` header on every request to scope rate-limit key `ip:192.0.2.42`), (b) fire (limit + 1) sequential POSTs with valid bodies, (c) assert the last response is 303 to `/?error=rate-limited`. Choose an email like `smoke-rl-${Date.now()}@example.com`. Note: the first N may succeed (creating the row) — that's fine; the assertion is only on the LAST response.

    Caveat for case 7: the rate-limit map is in-memory per server process. If the dev server was just restarted, the limiter is empty — case 7 reliably triggers. If the server has been running for a while and other smoke runs already populated it, the trigger might fire earlier than expected. Document this in the case comment; the assertion only requires that AFTER excess attempts, `rate-limited` is returned.
