<objective>
Land the canonical 48-team data file (`references/teams.json`) AND the slug↔football-data.org-ID lookup mechanism. Per CONTEXT.md D-06 the planner picks ONE of two paths; this plan picks **(b) runtime join — add a `slug` column to the `teams` SQLite table** because (i) it avoids a parallel committed file that can drift from the live ingestor and (ii) the kickoff cron already JOINs `teams` from `matches`, so the extra column adds zero JOIN cost.

Purpose: every consumer in Phase 5 (signup allow-list, kickoff cron JOIN, smoke tests) needs this data layer in place. Doing it in Wave 1 alongside the docs sweep keeps Wave 2 (vip_signups schema migration) and Wave 3 (signup + downstream rewrites) unblocked.

Output: a 48-element JSON array, a slug-aware `teams` SQLite schema, an updated ingestor, and a backfill script. Zero behavior change in the live signup or cron path yet — those flip in Waves 2 + 3.
