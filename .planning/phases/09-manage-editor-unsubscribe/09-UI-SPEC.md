## Scope Note

Phase 9 is a brownfield page redesign of `/manage`. The page becomes
dual-mode (signed-out magic-link form vs signed-in editor). The signed-in
editor branch is new and ports the body of the current `src/pages/schedule.astro`
(minus the 48-team checkbox grid, which becomes a single `<select>`).

**Visual scope:** `/manage` only. The Phase 6 consumer landing page at `/` uses a
different (light + sans) style block and is NOT touched here. `/manage` continues
the original Bitcoin-minimalist mono + dark surface aesthetic per CLAUDE.md
§Conventions ("aesthetic is 'Bitcoin minimalist on near-black'", "One mono font
everywhere"), which is also the aesthetic of the existing `src/pages/manage.astro`
and `src/pages/schedule.astro`. The `src/pages/index.astro` mention in CONTEXT D-03
refers to the `<select>` HTML structure (confederation `<optgroup>` grouping +
48 `<option>` elements), NOT the visual styling.

## Color

Single dark surface palette (Bitcoin-minimalist), inherited verbatim from `src/pages/schedule.astro:274-284` and `src/pages/manage.astro:85-94`:

| Role | Value | CSS Token | Usage |
|------|-------|-----------|-------|
| Dominant (60%) | `#0b0b0e` | `--bg` | Page background; the dominant near-black surface |
| Secondary (30%) | `#ececf1` | `--fg` | Body text (light on dark); the dominant readable text color |
| Secondary surface | `rgba(255, 255, 255, 0.06)` | `--surface` | Card / fieldset / match-row interior background (slight lift off the near-black) |
| Secondary line | `rgba(255, 255, 255, 0.18)` | `--line` | Hairline borders on fieldsets, inputs, match rows |
| Dim text | `#b8b8c2` | `--fg-dim` | Subhead, fineprint, dim metadata, muted labels |
| Accent (10%) | `hsl(18 70% 56%)` | `--accent` | See "Accent reserved for" below |
| Accent ink | `#0b0b0e` | `--accent-ink` | Text color on top of accent backgrounds (i.e. the submit button label) |
| OK semantic | `hsl(140 50% 60%)` | `--ok` | `.flash.ok` border + text (saved status) |
| Destructive / error semantic | `hsl(360 70% 65%)` | `--err` | `.flash.err` border + text (bad-token / bad-tz / bad-team / unknown / server) |

**60/30/10 split mapping:**
- 60% dominant `--bg` (`#0b0b0e`) — page background fills the viewport.
- 30% secondary — split across `--fg` (text on background), `--surface` (lifted card interiors), and `--line` (hairline borders). Together these are the structural mass of the editor.
- 10% accent `--accent` (`hsl(18 70% 56%)`) — reserved per the explicit list below.

**Accent reserved for (explicit list — never anywhere else on `/manage`):**
1. Banner pill `<span class="banner">` — border + text color (1px solid `--accent` + `color: var(--accent)`)
2. Save button background (`button[type="submit"]` — `background: var(--accent)`, ink in `--accent-ink`)
3. Logout `<button class="link-button">` text color
4. Hyperlinks `.link` and `.link-button` (text color `--accent`, underline-offset 3px)
5. Focused input border (`input:focus, select:focus { border-color: var(--accent) }`)
6. Team-tile TLA prefix `.team .tla` color — **N/A in Phase 9** (checkbox grid is replaced by `<select>`; no TLA prefix on `<option>` per RESEARCH Q6)
7. Focused textarea outline (`.feature-request textarea:focus` — outline `--accent`) — **N/A in Phase 9** (feature-request textarea is dropped per RESEARCH Open Q1; this rule is removed)

Accent is NEVER applied to: body text, subhead text, fieldset borders (those use `--line`), match-row borders, or background fills outside the submit button.

**Destructive color (`--err`):** reserved for `.flash.err` status messages (bad-token / bad-tz / bad-team / unknown / server). NOT used on the submit button — the submit button is always `--accent` even when re-saving from an error state, because the user is initiating a save (constructive), not confirming a delete.

**No destructive confirmation dialogs in Phase 9.** Unsubscribe is initiated via email link, not from this page; logout is reversible. See "Copywriting Contract" below.

## Interaction & State Contract

### Form interactions

1. **Team `<select>` change** — pure HTML, no JS handler. Submitted with the form. The server pre-renders `<option value={t.slug} selected={t.slug === user?.team}>` so the current selection is reflected on page load.
2. **TZ change** — existing `<script is:inline>` at `schedule.astro:207-269` is ported verbatim. Detected TZ from `Intl.DateTimeFormat().resolvedOptions().timeZone` is the default; the saved DB value (from `user.timezone`) takes precedence; user can click "change" to reveal the full IANA `<select>` populated from `Intl.supportedValuesOf('timeZone')`.
3. **Save** — submit fires standard HTML form POST to `/api/save-selection`. Server returns 303 → `/manage?status=<slug>` (one of `saved` / `bad-token` / `bad-tz` / `bad-team` / `unknown` / `server`). The server-rendered Astro page reads `?status=` in the frontmatter and renders the `.flash.ok` or `.flash.err` block accordingly. No client-side AJAX, no spinners, no optimistic updates.
4. **Logout** — submit fires standard HTML form POST to `/api/logout`. Server clears the session cookie and redirects to `/`. No client-side confirmation.

### Keyboard / focus order (top to bottom in DOM order)

1. (Skip target) — none; page is short enough that skip-to-main is not required, but the existing `<main>` element provides semantic landmark.
2. TZ "change" button (when present)
3. TZ `<select>` (when revealed by clicking "change")
4. Team `<select>` — the primary call-to-action input
5. Submit button (`Save selection`)
6. Footer back-link (`← oddlympics.app`)
7. Logout button

All interactive elements receive a visible focus indicator: `border-color: var(--accent)` on `<select>` and `<input>` per `schedule.astro:107`. The submit button's hover state (`filter: brightness(1.08)`) doubles as a visible affordance; its focus state inherits the browser default outline (executor MAY add an explicit `:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px }` rule to match the existing `.feature-request textarea:focus` pattern at `schedule.astro:342` — recommended but not required).

### ARIA contract

- `<main class="hero">` — implicit landmark; no `role` override.
- `.banner` pill — decorative, no `role`. Copy "Pick a team" or "Your schedule" is purely visual context.
- `.flash` status message — `role="status"` (verbatim from `schedule.astro:111`). Polite announcement on page load when `?status=saved` etc.
- Team `<select>` — has `aria-label="Your team"` (no visible `<label>` per the inline `.tz-row`-style layout); `required` attribute on the element provides the unset-state validation.
- TZ "change" button — `<button type="button">` (verbatim from `schedule.astro:120`); no `aria-expanded` needed since the toggle reveals the adjacent `<select>` immediately.
- Logout `<button class="link-button">` — inherits the `<form action="/api/logout">` wrapper; no extra ARIA.
- noscript fallback for TZ — `<noscript>` wraps a plain `<span>` carrying the disabled-JS explainer (verbatim from `schedule.astro:129-131`).

### Mobile breakpoint

Single breakpoint at `@media (max-width: 640px)` (verbatim from `schedule.astro:345`):
- `--pad` shrinks from 48px to 24px.
- `.headline` shrinks from 36px to 28px.
- `.match` grid collapses from 3-column to 1-column (`grid-template-columns: 1fr; gap: 4px`).
- The team `<select>` is already full-width on desktop, so it stays full-width on mobile with no extra rule.

The 520px breakpoint mentioned in the prompt is the `index.astro` consumer landing breakpoint (different page, different style block). For `/manage`, the established breakpoint is 640px — keep it.

### Reduced motion

Inherit any `prefers-reduced-motion` rules from the source page. The signed-in editor branch has minimal motion (no animated banner, no transitions besides the submit button hover filter and the optional `details summary` rotation on `index.astro`). No new animations introduced. If executor adds a hover transition on the submit button beyond what's already in `schedule.astro:316`, it MUST be wrapped in `@media (prefers-reduced-motion: no-preference)`.
