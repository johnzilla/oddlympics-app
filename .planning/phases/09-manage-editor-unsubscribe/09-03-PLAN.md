<objective>
Replace `src/pages/schedule.astro` (currently a full 351-line editor page)
with a thin server-rendered 301 redirect handler that sends every GET
`/schedule[?...]` to `/manage[?...]`, preserving the query string
verbatim. This implements CONTEXT D-01's URL consolidation: the editor
moves to `/manage` (Plan 09-04), and `/schedule` becomes a permanent
redirect so:

- In-flight `sendManageLink()` emails minted before Plan 09-02's URL
  change still route their `?token=` payload to the new editor within
  their 24h TTL window (RESEARCH Q2 verified).
- Bookmarks and inbound links to `/schedule` keep working.
- The redirect is 301 (permanent) so browsers cache it.

Per RESEARCH Q1, Astro's `redirects:` config in `astro.config.mjs` does
NOT preserve query strings (upstream limitation, GitHub issue #7890), so
this MUST be an in-process Astro page that reads `Astro.url.search`. The
file stays — it does not get deleted.

Purpose: Wave 2 first task because Plan 09-04 (the manage.astro editor)
removes the legacy session→/schedule redirect at manage.astro:7. If
09-04 ships before 09-03, then a session-valid user hits /manage → sees
editor → submits → 303 redirects work; if 09-03 ships before 09-04, a
session-valid user hits /manage → manage.astro:7 redirects to /schedule
→ schedule.astro 301 → back to /manage → manage.astro:7 redirects to
/schedule → infinite loop. Plan 09-04 MUST remove that line 7 redirect
BEFORE the editor branch lands; planner has ordered tasks within 09-04
to make removal the first step. Both 09-03 and 09-04 can land in any
order within Wave 2 SO LONG AS manage.astro:7 redirect is removed in
the same 09-04 commit — to avoid the race we explicitly mark 09-04 as
the "removal first, editor second" plan.

Depends on 09-02 because the schedule.astro redirect is only useful
once `sendManageLink()` is also writing the new URL (otherwise the
redirect-vs-direct-link split is asymmetric for current emails).

body / no styles. No other files modified.
