    ```
    npm run og:render-teams
    ```

    If exit code is non-zero:
    - Read stderr — the script logs which team and which check failed.
    - If a LAND-02 failure: a team label in `references/teams.json` accidentally matches a banned term. Audit the entry; if legitimate, the LAND-02 grep is too aggressive — pause and surface to user (do NOT loosen the grep without approval).
    - If a size > 300KB failure: the SVG render produced an unexpectedly large PNG. Likely cause: font fallback (resvg silently rendered with a system font because `loadSystemFonts: false` is missing or the `fontFiles` paths are wrong). Inspect the script — `loadSystemFonts: false` MUST be present; `references/fonts/{JetBrainsMono-Bold,Inter-Regular,Inter-Bold}.ttf` MUST exist.
    - If dimensions wrong: the SVG `viewBox` or `width`/`height` is off — should have been preserved from `og-image.svg`. Audit `references/og-image-team.svg`.
    - **Do NOT proceed to step B if the render exits non-zero. Fix the underlying issue first.**

    **B. Spot-check 3 PNGs for visual sanity:**

    1. `file public/og/england.png` — must report `PNG image data, 1200 x 630, 8-bit/color RGB, non-interlaced` (or similar PNG signature). NOT "data" or "empty file".
    2. `wc -c public/og/england.png public/og/usa.png public/og/bosnia.png` — each between ~20KB and 300KB.
    3. `ls public/og/ | wc -l` returns `48` (exactly 48 PNG files).
    4. `ls public/og/ | grep -c '\.png$'` returns `48`.
    5. **Long-label visual check (optional but recommended):** If `bosnia.png` exists (slug for Bosnia and Herzegovina), open it (e.g. `open public/og/bosnia.png` on macOS). The headline "Following Bosnia and Herzegovina." should fit on one line at 44pt without overflow. If it overflows, this is the signal to adjust y-offsets in `references/og-image-team.svg` (Plan 15-02 task 1) — but per the 15-PATTERNS.md adaptation note, this is acceptable for v1 and can be tuned post-render.

    Note: 15-CONTEXT.md <specifics> shows the slug `bosnia` in a sample; verify the actual slug in `references/teams.json` — it may be `bosnia_herzegovina` or similar. Use whatever the actual long-label-team slug is.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator dev machine → `public/og/*.png` | Operator runs `npm run og:render-teams` locally; output is committed to git |
| `public/og/*.png` → CDN/Caddy → user | PNGs served as static assets, visible publicly via `https://oddlympics.app/og/<slug>.png` |
| `public/og/*.png` → git history | PNGs become part of git blob storage forever (revert is non-destructive) |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

| T-15-12 | Tampering | A maintainer accidentally commits a tampered PNG (manual edit) | mitigate | `npm run og:render-teams` is deterministic (`loadSystemFonts: false` + vendored fonts + fixed SVG). Re-running on a clean checkout produces byte-identical output. A reviewer can `git diff --stat` and `npm run og:render-teams` again to verify. Documented in `scripts/render-team-og-images.mjs` header. |
| T-15-13 | Denial of Service | git repo bloat from 48 × ~150KB PNGs | accept | Expected total ~5-10MB added to git. Phase 8 committed 1 PNG (~80KB); this 48-x scale is the documented cost of the runway-safe build-time-render-and-commit pattern (D-09). Alternatives (render-on-request, S3 bucket, etc.) are explicitly out of scope per D-09. Repo is small (~50MB current); 10MB bump is non-issue. |
| T-15-14 | Repudiation | Stale PNGs (label changes in `references/teams.json` without re-render) | accept | Per D-09, "no CI gate enforces re-render; lives in the same trust contract as the existing `npm run og:render`". | T-15-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan — just invokes existing scripts. No legitimacy gate needed. |
