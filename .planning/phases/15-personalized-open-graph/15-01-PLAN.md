<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → `/r/[code]` route | Untrusted code path-segment crosses here; no auth on this route by design (D-02, D-03) |
| `/r/[code]` route → SQLite | Parameterized query via `lookupTeamByReferralCode.get(code)`; no string interpolation |
| `/r/[code]` route → response HTML | `code` may be echoed into `<a href="/?ref=CODE">`, `og:url`, JSON.stringified into `define:vars` bounce script |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering / XSS | `src/pages/r/[code].astro` — reflected XSS via `code` echoed into `<a href>`, `og:url`, or `bounceUrl` | mitigate | Code-shape gate `/^[a-z0-9]{8}$/` enforced BEFORE any echo; invalid shape → treat as unresolved (bounceUrl = '/', no code echoed). Astro JSX `href={bounceUrl}` auto-escapes; `define:vars` JSON-encodes the value into the script source. Combined: charset gate + auto-escape + JSON encoding = three independent layers. |
| T-15-02 | Tampering | `src/pages/r/[code].astro` — path traversal via `code = '../../../etc/passwd'` | mitigate | Same `/^[a-z0-9]{8}$/` gate rejects any value containing `/`, `.`, `..`, or control chars. Path is built via `resolve(repoRoot, 'public/og', row.team + '.png')` where `row.team` comes from the DB allow-list (48 slug values from `references/teams.json`), NOT from `code`. |
| T-15-03 | Spoofing | `src/pages/r/[code].astro` — open-redirect via `bounceUrl` | mitigate | `bounceUrl` is constructed from a hardcoded `'/?ref=' + code` or `'/'` template — never echoes external URL. The shape-gated `code` is `[a-z0-9]{8}`, cannot contain `://` or other URL constructs. No `Location:` header is set; bounce is purely client-side via meta-refresh + `location.replace`. |
| T-15-04 | Denial of Service | `/r/[code]` route — lookup spam | accept | The existing `src/lib/rate-limit.ts` middleware on `/api/*` does NOT cover `/r/*`. Per-request work is a single indexed SELECT + one `existsSync` (synchronous filesystem stat). At ~1ms per request on a $6 droplet, even sustained 1000-req/s spam is well within capacity. Documented as accepted; revisit if real abuse seen post-launch. |
| T-15-05 | Information Disclosure | `lookupTeamByReferralCode` — leaks email or status to unauth route | mitigate | New statement returns ONLY `{ referral_code, team }` — narrowed SELECT, no `email`, no `confirmed_at`, no `unsubscribed_at`. Documented in db.ts comment as intentional per D-03. |
| T-15-06 | Information Disclosure | `/r/[code]` route — confirms whether a code exists by varying response body | accept | Per D-02 / Phase 13 D-08, code lookups are intentionally observable (the carry-through into `referred_by` requires lookups to be possible). A bot enumerating codes to enumerate accounts learns only that a code maps to a team slug — no email, no status. Public referral codes have ~36^8 ≈ 2.8e12 keyspace; brute-force scraping is computationally moot and operationally noisy. |
| T-15-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan — only uses existing `astro`, `better-sqlite3`, `node:fs`, `node:path`, `node:url`. No legitimacy gate needed. |
</threat_model>
