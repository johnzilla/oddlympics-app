<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/signup POST | The `ref` form value is fully attacker-controlled (any string, any length) |
| submitted ref → vip_signups lookup | The ref is used in a prepared-statement SELECT against `referral_code` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-04 | Tampering | self-referral gaming (D-08) | mitigate | After resolving the ref, compare the resolved code-owner's email to `rawEmail` (both already trimmed+lowercased). On a match, `referred_by` stays NULL — a user cannot attribute a referral to themselves. Reachable only on a re-signup (a brand-new email owns no prior code). |
| T-13-05 | Denial of Service | malformed/oversized `ref` value (ASVS V5) | mitigate | `ref` is `.trim().toLowerCase()`d then used only as a parameterized SELECT bind value — no interpolation, no regex backtracking. A non-matching value resolves to `undefined` → `referred_by` NULL → signup continues. Bad ref never calls `back()`, never throws (D-08/D-09); the "signup never rejects" contract holds (SC4). |
| T-13-06 | Information Disclosure | referral-code enumeration via the signup endpoint (ASVS V5) | accept | The ref lookup is a read-only parameterized SELECT folded into the signup path; it is not a separate queryable endpoint and returns no data to the client (the response is an identical 303 whether the ref matched or not). The existing signup rate-limit (5/hr/IP + 5/hr/email) bounds enumeration attempts; the 2.8e12 keyspace makes a hit improbable. No new control needed. |
| T-13-07 | Spoofing | forged `ref` claiming attribution | accept | Referral codes are public identifiers, not credentials — attribution is intentionally lightweight (PROJECT.md decision). A forged-but-real ref simply attributes a signup to a real existing code; no resource is gated, no reward is granted (rewards are out of scope). ASVS V2/V3/V4 do not apply. |
| T-13-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced this plan — zero install surface. |
</threat_model>
