## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| boot-time migration → SQLite file | Migration mutates the on-disk DB schema; a malformed migration crashes the server on boot |
| generated code → vip_signups.referral_code | Codes become public identifiers shared in URLs; weak generation enables guessing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Denial of Service | `db.ts` migration block | mitigate | Additive-only migration (D-03): `pragma_table_info` probe + `ALTER TABLE ADD COLUMN`, no DROP COLUMN, no version assert. Re-running is a no-op. A boot-crash from an inline UNIQUE constraint is avoided by using `CREATE UNIQUE INDEX IF NOT EXISTS` separately (verified SQLite landmine). |
| T-13-02 | Information Disclosure | `referral.ts` code generation (ASVS V6) | mitigate | Codes generated with `node:crypto` `randomInt` (CSPRNG, bias-free) — not `Math.random`, not hand-rolled. 8-char `[a-z0-9]` keyspace ≈ 2.8e12; combined with the existing signup rate-limit (5/hr/IP) brute-force enumeration is impractical (D-01). |
| T-13-03 | Tampering | backfill loop collision handling | accept | A `SQLITE_CONSTRAINT_UNIQUE` collision is astronomically rare (~3.6e-13 per attempt at current row counts); the 5-attempt retry is a defensive precaution. Non-collision errors are re-thrown, not swallowed. |
| T-13-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced this plan — zero install surface. (RESEARCH.md § Package Legitimacy Audit: not applicable.) |
</threat_model>
