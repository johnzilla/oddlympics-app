<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL `?ref=` query param → page | Attacker-controlled; any visitor can craft `/?ref=anything` |
| localStorage `oddlympics_ref` entry → hidden field | Persists across visits; readable/writable by any same-origin script |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-08 | Tampering | localStorage poisoning of `oddlympics_ref` (ASVS V5) | mitigate | The client performs NO validation and grants the stored ref NO trust — it only transports the string into the hidden field. A poisoned/garbage stored value resolves server-side to `referred_by` NULL (Plan 13-02 lookup miss). The `JSON.parse` is wrapped in its own `try/catch`; a malformed entry is ignored, not crashed on (D-13). |
| T-13-09 | Denial of Service | malformed/oversized `?ref=` value breaking the page | mitigate | The entire reader block is `try/catch`-wrapped (D-13). Any exception (bad URL, localStorage `DOMException` in private mode, JSON error) falls through silently — the form still submits, the page still renders. No regex on the ref value, no unbounded loop. |
| T-13-10 | Information Disclosure | XSS via the `?ref=` value | mitigate | The ref value is only ever assigned to a form `input.value` via the DOM property (`refInput.value = activeRef`) — never `innerHTML`, never written into markup. Setting `.value` does not execute script. No new XSS surface. |
| T-13-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced this plan — zero install surface. No framework JS (plain inline script per CLAUDE.md). |
</threat_model>
