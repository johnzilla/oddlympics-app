## Trust Boundaries

| Boundary | Description |
|----------|-------------|

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-03 | Tampering | `team` slug values in `/api/save-selection` | mitigate | Every submitted slug validated against `VALID_TEAMS` (closed allow-list of 48); ANY unknown slug rejects the WHOLE submission with `bad-team` and writes zero rows (Task 1). |
| T-12-04 | Denial of Service | unbounded N team slugs (mass-insert) | mitigate | Server-side `validSlugs.length > 5` → `too-many` reject before any DB write; max 5 inserts per save bounds the write. Reject precedes the transaction (Task 1). |
| T-12-05 | Spoofing / Elevation | writing another user's `user_teams` | mitigate | Existing auth chain unchanged: `verifyToken(token,'manage')` (HMAC, timing-safe) OR `readSessionFromCookie`; writes keyed strictly to the authenticated `result.email`, never a form field. |
| T-12-06 | Tampering (SQLi) | slug values reaching SQLite | mitigate | All writes via `db.prepare`-backed `deleteUserTeams`/`insertUserTeam`/`updateTimezone` inside a single `db.transaction`; matches-query slug→id resolution uses `?` placeholders only — zero string interpolation (Task 1 + Task 2). |
| T-12-07 | Repudiation | which rule a save violated | accept (mitigated by logging) | `console.error` lines distinguish empty vs bad-slug (logged raw/valid arrays) vs too-many (logged count) per D-05; journald captures stdout/stderr (existing logging posture, no new logger). |
| T-12-08 | Information Disclosure | XSS via team label / status echo into HTML | mitigate | Labels come from the static `references/teams.json` allow-list (not user input); Astro auto-escapes `{...}` expressions; STATUS_COPY is a fixed server-side map keyed by a closed enum, not the raw query param. |
| T-12-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed in this plan. N/A — no install task. |
